# 图片棋盘的处理

## 问题背景

中国象棋项目支持两种棋盘样式：**woods** 和 **eleeye**，这两种样式在以下方面存在显著差异：

1. **棋子尺寸不同**：57x57 vs 34x34
2. **棋盘尺寸不同**：521x577 vs 362x425

## 样式对比

| 特性 | woods | eleeye |
|------|-------|--------|
| 棋子格式 | PNG（带透明背景） | BMP（无透明背景） |
| 棋子尺寸 | 57 x 57 像素 | 34 x 34 像素 |
| 棋盘格式 | JPG | JPG |
| 棋盘尺寸 | 521 x 577 像素 | 362 x 425 像素 |
| 单元格大小 | 57 像素 | 40 像素 |

## 为什么使用 PNG 而不是 JPG

**重要原因：透明背景支持**

- **PNG**：支持 alpha 通道透明背景，棋子边缘清晰平滑
- **JPG**：不支持透明背景，透明区域会被替换为某种颜色（通常是白色），导致：
  - 棋子边缘出现黑斑或白边
  - 黑色棋子的透明区域显示为半白色
  - 视觉效果严重受损

因此，**棋子必须使用 PNG 格式以保持透明背景效果**。

## 解决方案

### 1. 使用底层函数加载图片

由于 HBitmap 类在处理 PNG 格式时存在兼容性问题（特别是在 GTK 平台上），我们使用 HwGUI 提供的底层函数：

```harbour
function xq_LoadImageFile( par_cFileName )
   LOCAL hImg

   hImg := hwg_OpenImage( par_cFileName, .F. )
   IF Empty( hImg )
      OutErr( "hwg_OpenImage 失败: " + par_cFileName, hb_eol() )
   ENDIF

RETURN hImg
```

**底层函数的优势**：
- 跨平台兼容性更好
- 在 Linux/GTK 上支持 PNG、JPG、BMP、ICO 等多种格式
- 稳定可靠，不会出现段错误

### 2. 加载棋子图片

```harbour
static FUNCTION LoadPieceImage( par_cStyle, par_cFileName, par_aPieceImages, par_cKey )
   LOCAL hImg, aSize, cFilePath

   cFilePath := xq_GetSkinPath( par_cStyle, par_cFileName )
   hImg := xq_LoadImageFile( cFilePath )
   IF !Empty( hImg )
      aSize := hwg_Getbitmapsize( hImg )
      // 存储: { 图片句柄, 宽度, 高度 }
      par_aPieceImages[par_cKey] := { hImg, aSize[1], aSize[2] }
      OutErr( "棋子加载成功: " + par_cKey + " (" + Str(aSize[1]) + "x" + Str(aSize[2]) + ")", hb_eol() )
   ELSE
      OutErr( "棋子加载失败: " + par_cKey + " (" + cFilePath + ")", hb_eol() )
   ENDIF

RETURN .T.
```

#### 绘制时使用 hwg_DrawTransparentBitmap

```harbour
FOR i := 1 TO 10
   FOR j := 1 TO 9
      cPiece := aBoardPos[i,j]
      IF cPiece != "0" .AND. hb_hHasKey( aPieceImages, cPiece )
         aImgInfo := aPieceImages[cPiece]
         x := nOffsetX + (j-1) * nCellSize + Int( (nCellSize - aImgInfo[2]) / 2 )
         y := nOffsetY + (i-1) * nCellSize + Int( (nCellSize - aImgInfo[3]) / 2 )
         // aImgInfo = { 图片句柄, 宽度, 高度 }
         hwg_DrawTransparentBitmap( hDC, aImgInfo[1], x, y )
      ENDIF
   NEXT
NEXT
```

**关键点**：
- 使用 `hwg_DrawTransparentBitmap()` 绘制 PNG 图片
- PNG 的 alpha 通道会自动处理透明效果
- 不需要手动指定透明色参数

### 2. 不同棋盘尺寸的处理

```harbour
function xq_LoadResources( par_cStyle, par_oBoardBitmap, par_aPieceImages, ;
                          par_nBoardWidth, par_nBoardHeight, par_nCellSize )
   LOCAL hImg, aSize
   LOCAL cFilePath

   // 加载棋盘背景
   cFilePath := xq_GetSkinPath( par_cStyle, BOARD_FILE_NAME )
   hImg := xq_LoadImageFile( cFilePath )
   IF !Empty( hImg )
      par_oBoardBitmap := hImg
      OutErr( "棋盘加载成功: " + cFilePath, hb_eol() )
   ENDIF

   // 加载所有棋子...

   // 动态计算棋盘尺寸
   IF !Empty( par_oBoardBitmap )
      par_nBoardWidth := par_oBoardBitmap:nWidth      // 动态获取棋盘宽度
      par_nBoardHeight := par_oBoardBitmap:nHeight     // 动态获取棋盘高度
      par_nCellSize := Int( par_nBoardWidth / 9 )  // 计算单元格大小
      OutErr( "棋盘尺寸: " + Str(par_nBoardWidth) + "x" + Str(par_nBoardHeight) + ", 单元格: " + Str(par_nCellSize), hb_eol() )
   ELSE
      par_nBoardWidth := 540
      par_nBoardHeight := 600
      par_nCellSize := 60
      OutErr( "棋盘加载失败，使用默认尺寸", hb_eol() )
   ENDIF

RETURN .T.
```

**关键点**：
- 不假设固定尺寸，所有尺寸都从图片文件动态读取
- 单元格大小 = 棋盘宽度 / 9

### 3. 不同棋子尺寸的居中绘制

```harbour
// 绘制时
aImgInfo := aPieceImages[cPiece]  // { hImg, width, height, transColor }

// 计算居中位置
x := nOffsetX + (j-1) * nCellSize + Int( (nCellSize - aImgInfo[1]:nWidth) / 2 )
y := nOffsetY + (i-1) * nCellSize + Int( (nCellSize - aImgInfo[1]:nHeight) / 2 )

// 使用 HBitmap:Draw() 方法绘制
aImgInfo[1]:Draw( hDC, x, y )
```

**居中计算示例**：

**woods 样式**：
- 单元格大小：57 像素
- 棋子大小：57 x 57 像素
- 偏移量：Int((57 - 57) / 2) = 0，刚好填满

**eleeye 样式**：
- 单元格大小：40 像素
- 棋子大小：34 x 34 像素
- 偏移量：Int((40 - 34) / 2) = 3，居中显示

## 核心设计思路

1. **不假设固定尺寸**：所有尺寸都从图片文件动态读取
2. **存储完整信息**：每个棋子存储 `{图片句柄, 宽度, 高度, 透明色}`
3. **自动计算居中**：使用 `(单元格大小 - 棋子大小) / 2` 自动居中
4. **格式自适应**：根据扩展名自动选择正确的透明色处理方式

## 平台差异处理

### 底层函数跨平台支持

```harbour
hImg := hwg_OpenImage( par_cFileName, .F. )
```

**支持的图片格式**：
- Windows: **bmp, jpg**
- Linux: **bmp, jpg, png, ico**

**说明**：
- `hwg_OpenImage()` 自动处理平台差异
- 在 Linux/GTK 上支持 PNG 格式及其 alpha 通道透明效果
- 在 Windows 上，如果需要支持 PNG，需要使用 GDI+ (`hwg_GdiplusOpenImage`)

## 总结

通过使用 HwGUI 底层函数、动态读取图片尺寸、存储完整的图片信息、自动计算居中位置，程序能够无缝支持不同尺寸的棋盘样式，无需针对每种样式编写特殊代码。这种设计具有良好的扩展性，将来添加新样式时只需提供符合约定的图片资源即可。

**技术要点**：
1. **图片格式**：棋子使用 PNG 格式以保持透明背景效果
2. **加载函数**：使用 `hwg_OpenImage()` 底层函数加载图片
3. **绘制函数**：使用 `hwg_DrawTransparentBitmap()` 绘制，自动处理 PNG 的 alpha 通道
4. **尺寸获取**：使用 `hwg_Getbitmapsize()` 获取图片尺寸
5. **跨平台**：底层函数在 Linux/GTK 上支持 PNG，Windows 上可选择性使用 GDI+
