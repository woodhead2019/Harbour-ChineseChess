# 象棋引擎说明

## 概述

本项目集成了两种中国象棋引擎接口，支持与外部引擎进行通信以提供AI对战功能。

## 支持的引擎

### 1. ElephantEye (象眼) - UCCI 协议

**协议**: UCCI (Universal Chinese Chess Interface) - 中国象棋专用协议

**路径**: `engines/eleeye/eleeye/eleeye`

**特点**:
- 开源的传统强力引擎
- 支持 UCCI 3.0 协议
- 使用位行和位列技术
- 支持静态局面搜索
- 支持循环着法检测

### 2. Pikafish - UCI 协议

**协议**: UCI (Universal Chess Interface) - 国际象棋标准协议（兼容中国象棋）

**路径**: `engines/Pikafish/src/pikafish`

**特点**:
- 基于 Stockfish 的现代引擎
- 使用 NNUE 神经网络评估
- 棋力强劲
- 支持位棋盘（Bitboard）表示
- 兼容 UCI 协议标准

## 协议说明

### UCCI (Universal Chinese Chess Interface)

UCCI 是专门为中国象棋设计的通信协议，由中国象棋开发者社区制定。

**主要特点**:
- 专为象棋优化
- 支持中国象棋特有功能（如开局库、残局库）
- 命令格式简单直观

**常用命令**:
```
ucci              # 握手命令
position fen <fen> # 设置局面
go time <ms>      # 开始思考（时间控制）
stop              # 停止思考
quit              # 退出引擎
```

### UCI (Universal Chess Interface)

UCI 是国际象棋的标准通信协议，由国际象棋引擎社区制定。Pikafish 将其适配到中国象棋。

**主要特点**:
- 国际标准，兼容性好
- 支持高级特性（如多重PV、深度控制）
- 模块化设计

**常用命令**:
```
uci               # 握手命令
position fen <fen> # 设置局面
go depth <n>      # 开始思考（深度控制）
stop              # 停止思考
quit              # 退出引擎
```

## ICCS 坐标格式

ICCS (Internet Chinese Chess Server) 坐标格式用于表示移动：

- 列: a-i (0-based, a=0, b=1, ..., i=8)
- 行: 0-9 (从下到上, 0=红方底线, 9=黑方底线)
- 移动格式: `<from_col><from_row><to_col><to_row>`
- 示例: `h2e2` 表示从 h2 移动到 e2

**注意**: ICCS 格式在 UCCI 和 UCI 协议中都使用。

## FEN 格式

FEN (Forsyth-Edwards Notation) 格式用于表示棋局状态：

```
rnbakabnr/9/1c5c1/p1p1p1p1p/9/9/P1P1P1P1P/1C5C1/9/RNBAKABNR w - - 0 1
```

格式: `<棋盘> <回合> <易位> <吃过路兵> <半回合计数> <全回合计数>`

**字段说明**:
1. **棋盘位置**: 从黑方底线到红方底线，用/分隔，数字表示连续空格
2. **当前回合**: `w`=红方走，`b`=黑方走
3. **王车易位**: 中国象棋不使用，固定为`-`
4. **吃过路兵**: 中国象棋不使用，固定为`-`
5. **半回合计数**: 从上次吃子或走兵以来的半回合计数（用于50回合规则）
6. **回合计数**: 完整的回合计数（红方走一次+黑方走一次=1回合），从1开始

**50回合规则**: 当半回合计数达到100时（即50回合无吃子无动兵），判和棋。

**回合计数规则**: 黑方走完才+1（每完整回合+1）。

**注意**: FEN 格式在 UCCI 和 UCI 协议中都使用。

## 注意事项

1. **数组格式差异**: 项目中存在两种棋盘表示方式：
   - `XQ_BOARD_SIZE` (90) - 10x9 棋盘，用于 FEN 转换
   - `XQ_ARRAY_WIDTH` (12) - 11x12 带边界数组，用于引擎内部计算

2. **FEN 转换**: `xq_BoardToFen` 函数期望 90 元素的数组，而不是 11x12 的数组

3. **引擎通信**: 当前实现使用管道通信（stdin/stdout）与引擎进程交互

4. **进程管理**: 窗口关闭时会自动清理引擎进程，防止进程泄漏

## 当前 UCCI 接口处理分析

### 当前已实现的应答处理

基于 `ucci_engine.log` 的引擎应答分析，当前 UCCI 接口程序（`xq_ucci.prg`）已正确处理以下应答：

1. **初始化握手**
   - `ucciok` - 引擎握手成功
   - `readyok` - 引擎准备就绪

2. **最佳走法**
   - `bestmove <move>` - 引擎推荐的最佳走法
   - `nobestmove` - 无走法（游戏结束/逼和/困毙）

3. **错误处理**
   - 参数验证（`par_hConfig` 哈希表检查）
   - 超时处理（30秒默认超时）
   - 停止请求处理（`stop` 命令）

### 未充分利用的引擎应答

#### 1. 引擎信息（初始化时）

**引擎应答示例**：
```
< id name ElephantEye | id version 3.31 | id copyright 2004-2016 www.xqbase.com | id author ElephantEye Development Team | id user ElephantEye Test Team | option usemillisec type check default true | option promotion type check default false | option batch type check default false | option debug type check default false | option ponder type check default false | option usehash type check default true | option usebook type check default true | option bookfiles type string default /home/woodhead/cchess/engines/eleeye/eleeye/BOOK.DAT | option hashsize type spin min 16 max 1024 default 16 | option idle type combo var none var small var medium var large default none | option pruning type combo var none var small var medium var large default large | option knowledge type combo var none var small var medium var large default large | option randomness type combo var none var tiny var small var medium var large var huge default none | option newgame type button | ucciok |
```

**当前处理**：只等待 `ucciok`，忽略所有 `id` 和 `option` 信息

**改进建议**：
- 解析并存储引擎名称、版本、作者信息
- 解析引擎支持的选项（hashsize、bookfiles、pruning、knowledge 等）
- 根据引擎能力自动优化配置
- 提供引擎信息查询接口

#### 2. 搜索信息（走棋计算时）

**引擎应答示例**：
```
< info time 1647 nodes 5912057 | info depth 10 score -3 pv b9c7 h0g2 i9i8 i0h0 i8f8 e3e4 h9i7 g2e3 h7e7 h0h5 |
< info depth 11 |
< info time 5005 nodes 18130571 | bestmove b9c7 ponder h0g2 |
```

**当前处理**：只记录日志，不解析这些信息

**改进建议**：
- **解析评分** (`score`) - 显示引擎的评估分数（负数对黑方有利）
- **解析 PV** (`pv`) - 显示引擎推荐的走法序列
- **解析搜索深度** (`depth`) - 监控引擎搜索深度
- **解析节点数** (`nodes`) - 监控计算量
- **解析思考时间** (`time`) - 监控实际思考时间
- **解析 ponder** - 预测对手的应着（用于异步思考）

#### 3. 特殊局面识别

**引擎应答示例**：
```
< info time 0 nodes 39 | info depth 1 score 9999 pv g7g0 | bestmove g7g0 |
```

**当前处理**：当作正常走法处理

**改进建议**：
- 识别极端分数（如 `score 9999`）- 可能是绝杀或特殊局面
- 在日志中标注特殊局面
- 可能需要特殊处理（如立即接受或警告用户）

#### 2. 添加搜索信息解析函数

```prg
function xq_ParseSearchInfo( cInfoLine )
   LOCAL nScore := NIL
   LOCAL cPV := ""
   LOCAL nDepth := NIL
   LOCAL nNodes := NIL
   LOCAL nTime := NIL
   
   // 解析评分
   IF "score" $ cInfoLine
      nScore := Val(ExtractValue(cInfoLine, "score"))
   ENDIF
   
   // 解析 PV
   IF "pv" $ cInfoLine
      cPV := ExtractPV(cInfoLine)
   ENDIF
   
   // 解析深度
   IF "depth" $ cInfoLine
      nDepth := Val(ExtractValue(cInfoLine, "depth"))
   ENDIF
   
   // 返回哈希表
RETURN { "score" => nScore, "pv" => cPV, "depth" => nDepth, "nodes" => nNodes, "time" => nTime }
```
